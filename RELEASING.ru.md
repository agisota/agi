# Выпуск релизов

В этом проекте для автоматического версионирования и управления релизами используется [changesets](https://github.com/changesets/changesets). Релизы распространяются по двум каналам:

1. **npm-пакеты** — публикуются автоматически через `version.yml` с помощью changesets
2. **GitHub Release с бинарными файлами для платформ** — собираются и загружаются через `release.yml` при пуше тега версии

## Как это работает

### 1. Добавьте changeset

Когда вы вносите изменение, которое должно попасть в релиз, добавьте changeset:

```bash
pnpm changeset
```

При этом вам будет предложено:
- Выбрать, какие пакеты затронуты
- Выбрать тип semver-инкремента (patch, minor, major)
- Написать краткое описание изменения

В каталоге `.changeset/` будет создан markdown-файл. Закоммитьте этот файл вместе с изменениями кода.

### 2. Version PR создаётся автоматически

Когда changeset-файлы попадают в `main` через слияние, рабочий процесс `version.yml` автоматически открывает (или обновляет) pull request **«Version Packages»**. Этот PR:

- Поглощает все ожидающие changeset-файлы
- Повышает версии пакетов согласно объявлениям в changeset-файлах
- Генерирует/обновляет файлы `CHANGELOG.md` для затронутых пакетов

### 3. Слейте Version PR для выпуска релиза

Когда вы выполняете слияние Version Packages PR:

- Рабочий процесс `version.yml` обнаруживает, что все changeset-файлы поглощены
- Он собирает все пакеты и публикует их в **npm** с аттестацией происхождения (provenance attestation)
- Он создаёт git-тег `v{version}` на основе версии CLI-пакета `kb`
- Пуш тега запускает `release.yml`, который:
  - Собирает бинарные файлы для конкретных платформ: Linux x64, macOS x64, macOS arm64 и Windows x64
  - Подписывает бинарные файлы macOS (codesign + нотаризация) и Windows (Authenticode)
  - Генерирует контрольные суммы SHA256 для всех бинарных файлов
  - Создаёт **GitHub Release** со всеми прикреплёнными бинарными файлами и контрольными суммами

## Каналы релизов

| Канал | Рабочий процесс | Триггер | Результат |
|---------|----------|---------|--------|
| npm | `version.yml` | Пуш в `main` | npm-пакеты с provenance |
| GitHub Release | `release.yml` | Тег версии (`v*`) | Подписанные бинарные файлы для платформ + контрольные суммы |

## Бинарные файлы для платформ

| Платформа | Имя бинарного файла | Подписан |
|----------|------------|--------|
| Linux x64 | `fusion-linux-x64` | — |
| macOS arm64 | `fusion-darwin-arm64` | ✓ (codesign + нотаризация) |
| Windows x64 | `fusion-windows-x64.exe` | ✓ (Authenticode) |

> macOS Intel (`darwin-x64`) намеренно не поставляется: CLI работает только на Apple Silicon, потому что GitHub-раннеры `macos-13` слишком дефицитны, чтобы собирать надёжно. Десктопный macOS DMG/ZIP остаётся универсальным.

## Тестирование сборок бинарных файлов

Используйте рабочий процесс **Test Release** (`test-release.yml`), чтобы вручную протестировать сборки бинарных файлов без создания настоящего релиза:

1. Перейдите в **Actions** → **Test Release** → **Run workflow**
2. Рабочий процесс собирает все 4 бинарных файла для платформ, запускает smoke-тесты и загружает артефакты
3. Скачайте артефакт `all-binaries`, чтобы изучить результат

## Ручной релиз (запасной вариант)

Если вам нужно выпустить релиз вручную, вы всё ещё можете запушить тег версии напрямую:

```bash
git tag v0.2.0
git push origin v0.2.0
```

Это запустит `release.yml` для сборки бинарных файлов и создания GitHub Release. Обратите внимание: публикация в npm обрабатывается отдельно через `version.yml` и не будет запущена ручным пушем тега.

## Доступные скрипты

| Скрипт | Описание |
|--------|-------------|
| `pnpm changeset` | Добавить новый changeset |
| `pnpm changeset status` | Проверить ожидающие changeset-файлы |
| `pnpm release` | Локальный интерактивный релиз: показывает changeset-файлы, позволяет принять или переопределить предложенную версию, затем повышает версию + собирает + публикует + создаёт теги |
| `pnpm release --yes` | То же самое, но автоматически принимает предложенную версию и пропускает финальное подтверждение |
| `pnpm release --dry-run` | Только предпросмотр — показывает changeset-файлы, предложенную версию и запрашивает переопределение, затем завершается до любых изменений файлов/git/npm |
| `pnpm release:version` | Применить changeset-файлы и повысить версии (используется CI) |
| `pnpm --filter @runfusion/fusion build:exe` | Собрать бинарный файл для текущей платформы |
| `pnpm --filter @runfusion/fusion build:exe -- --target <target>` | Кросс-компиляция для конкретной платформы |
| `pnpm --filter @runfusion/fusion build:exe:all` | Собрать бинарные файлы для всех платформ |

## Советы

- Каждое изменение, видимое пользователю, должно иметь changeset — CI напомнит вам, если он отсутствует
- Вы можете добавить несколько changeset-файлов на один PR, если вносите изменения в несколько пакетов
- Changeset-файлы автоматически удаляются при повышении версий
- CI проверяет компиляцию бинарных файлов при каждом пуше/PR, чтобы рано выявлять регрессии сборки
- Если в вашем проекте включён `completionDocumentationMode: "changeset"`, спецификации триажа будут явно требовать артефакты завершения `.changeset/*.md` для соответствующих задач; держите это согласованным с конвенцией релизов вашего репозитория.

## Внутренние пакеты

Следующие пакеты являются **внутренними** и **не публикуются в npm**:

- `@fusion/core` — основная доменная модель и хранилище задач
- `@fusion/dashboard` — веб-интерфейс и API-сервер
- `@fusion/engine` — ИИ-агенты и оркестрация
- `@fusion/plugin-sdk` — SDK для разработки плагинов
- `@fusion-plugin-examples/*` — примеры плагинов

У этих пакетов в `package.json` указано `private: true`, и они перечислены в массиве `ignore` файла `.changeset/config.json`, чтобы предотвратить случайную публикацию. В npm публикуется только пакет `@runfusion/fusion`.
