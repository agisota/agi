# Концепции

> Русский перевод [CONCEPTS.md](./CONCEPTS.md).

Общий доменный словарь этого проекта — сущности, именованные процессы и концепции статусов со специфичным для проекта значением. Изначально наполнен ключевым доменным словарём, затем пополняется по мере накопления выводов из процессов ce-compound и ce-compound-refresh; прямые правки допустимы. Это только глоссарий, а не спецификация и не свалка.

## Настройки и локализация

### Surface (поверхность)
Один из обращённых к пользователю фронтендов agi — браузерный дашборд и терминальный TUI. Поверхности имеют независимые рантаймы и стеки рендеринга, но должны разделять состояние уровня пользователя: настройка, изменённая на одной поверхности (тема, язык), переносится на другую.

### Global Settings (глобальные настройки)
Настройки уровня пользователя, сохраняемые на стороне сервера, которые применяются ко всем поверхностям и всем проектам — в противоположность настройкам уровня проекта. Значения валидируются на границе записи: невалидное значение отбрасывается, а не сохраняется, поэтому каждый читатель может доверять тому, что загружает.

### Workflow Setting (настройка рабочего процесса)
Типизированная настройка, объявленная рабочим процессом в его IR (id, тип, значение по умолчанию, опции), повторяющая форму пользовательского поля задачи. Объявления описывают схему; *значения* сохраняются по паре рабочий процесс + проект через единый валидирующий авторитет хранилища, поэтому встроенные рабочие процессы могут нести значения без редактируемости их IR. Движок потребляет **эффективные настройки** — сохранённое значение с откатом к значению по умолчанию из объявления, причём значения, которые больше не проходят валидацию против текущего объявления, отбрасываются (никогда не передаются в исполнение).

### Effective Settings (эффективные настройки)
Плоская карта значений формы `Partial<Settings>` на каждую задачу, которую движок читает на входе в executor, составленная из разрешённого рабочего процесса задачи: для каждой объявленной Workflow Setting сохранённое значение `(workflowId, projectId)` откатывается к значению по умолчанию из объявления, причём сохранённые значения, которые больше не проходят валидацию против текущего объявления, отбрасываются. Разрешение никогда не выбрасывает исключение — отсутствующий или повреждённый рабочий процесс деградирует до встроенных объявлений для кодинга — поэтому каждый сайт чтения получает пригодное значение. Поскольку значения по умолчанию во встроенных объявлениях байт-в-байт равны устаревшим значениям по умолчанию настроек проекта, ненастроенный проект разрешается в идентичное поведение через жёсткий перенос настроек.

### Moved Settings Keys (перемещённые ключи настроек)
Allowlist-надгробие (`MOVED_SETTINGS_KEYS`) ключей пошагового исполнения, ревью/одобрения и модельных линий по фазам, которые единоразовая миграция жёсткого переноса переместила из настроек проекта/глобальных в Workflow Settings. Это единственная запись о старых именах, и она защищает каждую поверхность, способную столкнуться с устаревшим payload — diff-ы кросс-нодовой синхронизации, импорты настроек v1 и устаревшие писатели — от воскрешения перемещённого ключа. Тест на консистентность гарантирует, что ключ живёт ровно в одном режиме (настройки проекта *или* список-надгробие, но никогда в обоих).

### Three-Tier Setting (трёхуровневая настройка)
Именованный паттерн персистентности для пользовательского предпочтения на дашборде: локальный для устройства кеш для мгновенного чтения, сквозная запись в Global Settings, чтобы другие поверхности её увидели, и гидрация при монтировании с сервера, когда локального значения нет. Локальный или незавершённый пользовательский выбор всегда побеждает серверную гидрацию, а изменения распространяются на другие открытые вкладки.

### Translation Placeholder (плейсхолдер перевода)
Значение пустой строки для ключа каталога в неанглийской локали, помечающее «ещё не переведено». Плейсхолдеры намеренно дозаполняются при добавлении ключей; в рантайме они трактуются как отсутствующие (никогда не рендерятся), откатываясь по цепочке локалей к английскому. Непустое значение — даже английское, оставленное в неанглийском каталоге — рендерится как есть.

### Supported Locale (поддерживаемая локаль)
Языковой тег из закрытого набора, для которого agi поставляет переводы. Любой внешний тег (браузера, окружения, флага) нормализуется в этот набор или отклоняется — никогда не пропускается сырым. Китайские теги маршрутизируются по письменности и региону, поэтому пользователям традиционной письменности никогда молча не подсовывают упрощённую, а два китайских варианта никогда не схлопываются в общий базовый тег.

## Миссии

### Отношения

Миссия владеет упорядоченным списком вех; веха владеет упорядоченным списком срезов; срез владеет набором функций. Статус прокатывается **вверх**, а не вниз: статус среза выводится из его функций, статус вехи — из её срезов, а статус миссии — из её вех. Autopilot действует на границе среза — он продвигает миссию, активируя следующий срез, как только текущий завершён.

### Mission (миссия)
Единица автономной многошаговой работы, которую система планирует, а затем самостоятельно доводит до завершения, декомпозируя на вехи. Миссия может выполняться под Autopilot или продвигаться вручную.

### Milestone (веха)
Упорядоченная фаза миссии, содержащая срезы и опционально зависящая от более ранних вех. Веха завершена только тогда, когда завершены все её срезы.

### Slice (срез)
Вертикально ограниченный, независимо завершаемый кусок вехи, содержащий функции. Статус среза выводится из его функций и достигает значения *complete* только тогда, когда каждая функция считается выполненной — что, для функции, несущей Contract Assertions, требует прошедшего Validator Run.

### Feature (функция)
Наименьшая единица работы миссии: единый поставляемый результат, оцениваемый против его Contract Assertions. Функция несёт как статус на доске (её колонку в рабочем процессе, например done), так и состояние цикла (её фазу исполнения); эти два различны и могут законно расходиться в полёте, но функция done, которая так и не достигла терминального состояния цикла — это нарушение инварианта, которое застопорит её срез.

### Fix Feature (функция исправления)
Функция, автоматически сгенерированная из проваленного Validator Run, чтобы нести работу по устранению для проваленных assertions, со ссылкой обратно на функцию, от которой она происходит.

## Проекты

### Project (проект)
Зарегистрированное рабочее пространство, над которым agi может оперировать: оно имеет канонический локальный путь, настройки и данные в рамках проекта и должно быть подкреплено пригодным к работе git-рабочим деревом, прежде чем исполнение задач сможет создавать из него рабочие деревья.

**Workspace** (рабочее пространство) — это особый вариант проекта, где зарегистрированный путь сам по себе не является git-репозиторием, но содержит несколько git-репозиториев как прямые поддиректории. agi обнаруживает суб-репозитории во время инициализации и записывает их в `.fusion/workspace.json`. В режиме workspace исполнение задач не требует единого рабочего дерева корневого уровня; вместо этого агент приобретает рабочие деревья по каждому репозиторию по требованию через `fn_acquire_repo_worktree`.

Слияния задач workspace **неатомарны**: каждый суб-репозиторий приземляется на свой собственный локальный интеграционный ref независимо, поэтому окно частичного приземления (часть суб-репозиториев слита, другие — нет) возможно в середине задачи — это состояние локально и сбрасываемо оператором.

### Project Identity (идентичность проекта)
Долговечная идентичность, которую зарегистрированный проект несёт локально, чтобы его можно было повторно прикрепить к центральному реестру после потери или перестроения центрального состояния, сохраняя строки, ключованные по тому же id проекта, вместо создания замены.

### Project Registration (регистрация проекта)
Процесс, который создаёт или повторно прикрепляет проект в центральном реестре agi для локального пути. Регистрация — это граница готовности: она должна завершаться только тогда, когда путь удовлетворяет инвариантам, от которых позже зависит исполнение, например является git-рабочим деревом.

## Исполнение миссии

### Autopilot
Именованный процесс, который наблюдает за активной миссией и продвигает её — активируя следующий ожидающий срез, как только текущий срез завершается — отслеживая при этом собственный жизненный цикл наблюдения/активации и обрабатывая повторы. Когда Autopilot не наблюдает за миссией, продвижение срезов откатывается на путь совместимости.

### Contract Assertion (контрактное утверждение)
Проверяемый критерий приёмки, связанный с функцией, который ИИ-валидатор оценивает, чтобы решить, действительно ли функция выполнена. Каждое утверждение несёт `type` (`static` или `behavioral`). Статические утверждения оцениваются инспекцией только на чтение. Поведенческие/багфиксовые утверждения занимают **состязательную позицию по умолчанию-к-провалу**: pass от судьи носит рекомендательный характер, и утверждение удовлетворено только тогда, когда поведенческий прогон верификации подтверждает наблюдаемый результат, прогоняя код. Каждая функция оценивается валидатором — функции, у которой отсутствует утверждение, оно лениво привязывается перед валидацией — и она засчитывается в завершение среза только после прошедшего Validator Run.

### Validator Run (прогон валидатора)
Единое исполнение, которое оценивает Contract Assertions функции и выдаёт исход pass, fail, blocked, error или **inconclusive**. Оно состоит из двух частей. **ИИ-судья только на чтение** инспектирует реализацию и записывает рекомендательный вердикт, не создавая задачи на доске и не редактируя код. Для поведенческих/багфиксовых утверждений отдельный **прогон верификации** затем подтверждает (или опровергает) судью, прогоняя код — поэтому Validator Run больше не является чисто read-only/статическим. Прогон верификации по-прежнему **не мутирует состояние миссии/доски**: он исполняется против изолирующей песочницы (fail-closed, когда таковой нет) и одноразового checkout на доверенной ревизии, не создаёт задачи на доске, не мутирует ни строку миссии/доски и оставляет исходное дерево git-чистым. Вердикт `inconclusive` (верификация не смогла запуститься или прийти к заключению) является первоклассным и отличается от `fail`: он маршрутизируется в needs-attention и не порождает Fix Feature. Прогон, оставленный работающим после исчезновения его владельца, пожинается в терминальное состояние error; пожатая функция done без задачи переводится восстановлением заново к терминальному вердикту, а не дедлочит срез.

### loop state (состояние цикла)
Позиция функции в цикле исполнения (реализуется, ожидает или проходит валидацию, ожидает исправления, прошла или заблокирована), отличная от её статуса на доске. Логика, которая гейтится на состоянии цикла, должна трактовать его как возможно устаревшее и возможно противоречащее статусу — функция может быть помечена done, тогда как её состояние цикла так и не было продвинуто дальше implementing.

## Жизненный цикл слияния

### Task (задача)
Ключевая сущность доски: единица работы, которая движется через колонки (triage, todo, in-progress, in-review, done, archived) и исполняется агентами. Задача несёт собственные настройки на уровне задачи, которые могут переопределять значения по умолчанию уровня проекта.

### Workflow Runtime (рантайм рабочего процесса)
Авторитетный рантайм жизненного цикла задачи. Он разрешает задачу в IR рабочего процесса, обходит граф, маршрутизирует исходы узлов и вызывает примитивы рантайма для побочных эффектов. Субстрат движка по-прежнему владеет планированием, маршрутизацией claim-ов, персистентностью, конкурентностью, супервизией процессов, хранилищем и обвязкой аудита; политика жизненного цикла живёт в узлах рабочего процесса и встроенном IR рабочего процесса.

### Engine Singleton Lock (блокировка синглтона движка)
Гарантия взаимного исключения на каждую машину, обеспечивающая, что только один процесс agi запускает движок для данного проекта, комбинируя lockfile в директории `.fusion/` проекта с loopback-сокетом на каждый проект. Неудача в приобретении (`EngineAlreadyRunningError`) — это **позитивное доказательство того, что движок уже запущен** для этого проекта где-то ещё на машине — а не ошибка, которую следует проглотить, и не «нет движка». Процесс, которому отказано в блокировке, удерживает это как факт: он сообщает о движке как о доступном (чтобы поверхности UI не утверждали, что он недоступен), пока реконсиляция продолжает повторять попытки, так что он перехватывает управление, если текущий владелец завершается.

### ACP Ask Path (путь запроса ACP)
Однооборотный запрос к модели только на чтение, маршрутизируемый через рантайм ACP, а не через режим печати CLI. Раннер аккумулирует стримящуюся прозу, может восстановить завершающий JSON-объект для структурированных швов и трактует аномальные причины остановки ACP как неполные ответы для использования валидатором.

### Claude Bridge (мост Claude)
Запиненный подпроцесс `claude-code-cli-acp`, поставляемый с плагином рантайма ACP. Он говорит на ACP по stdio с agi, управляя при этом реальным интерактивным `claude` через PTY, и разрешается из принадлежащего плагину дерева `node_modules`, а не из PATH.

### Runtime Primitive (примитив рантайма)
Именованная, инъецированная операция, которую узел рабочего процесса может вызвать для выполнения побочных эффектов без зависимости от ветвей жизненного цикла `executor.ts`. Примеры включают сессию планирования, сессию кодинга, исполнение/сброс шага, ревью, верификацию, шаг рабочего процесса, переход, запрос слияния, прерывание и аудит. Примитивы — это граница между политикой рабочего процесса и субстратом движка.

### Built-in Lifecycle Node (встроенный узел жизненного цикла)
Узел во встроенном рабочем процессе, который выражает поведение agi по умолчанию, такое как планирование, execute, ревью, merge, parse-steps, step-review или действия жизненного цикла PR. Встроенные узлы жизненного цикла — это слой совместимости для существующего поведения: изменение исполнения по умолчанию означает изменение встроенного рабочего процесса и его обвязки примитивов, а не добавление скрытых императивных ветвей.

### Recovery Event (событие восстановления)
Наблюдаемое рабочим процессом условие, требующее политики восстановления, такое как неполная реализация, недоступность ревью, таймаут слияния, требуемое ручное слияние, конфликт интеграции или жёсткая отмена. Восстановление может использовать примитивы движка для прерывания процессов, записи записей аудита, сброса шагов или паркования задач, но решение о маршрутизации принадлежит логике рабочего процесса.

### Auto-merge (авто-слияние)
Именованный процесс, который автоматически приземляет ветку завершённой задачи на её цель слияния, как только задача достигает In-review и проходит свои блокировщики слияния. Гейтится дважды: настройка уровня проекта включает его глобально, и каждая задача может нести явное переопределение на уровне задачи.

Переопределение на уровне задачи имеет приоритет в обоих направлениях: явное включение на уровне задачи продолжается даже когда глобальная настройка выключена, а явное отключение на уровне задачи маршрутизирует слияние в Manual-required даже когда глобальная настройка включена. Гейты на уровне триггеров (постановка в очередь, проходы Self-healing) должны оценивать аддитивно — глобальное «вкл» пропускает всё для нижестоящей маршрутизации; глобальное «выкл» допускает только явные включения на уровне задачи — а не схлопывать переопределение в единое эффективное значение, что обескровило бы маршрутизацию Manual-required.

### In-review (на ревью)
Колонка статуса задачи между исполнением и завершением: работа выполнена, и ветка ожидает слияния. Задача In-review либо авто-сливается, либо ждёт ручного слияния человеком (флоу на основе PR / ручной), либо выдаёт диагностику застоя, когда она простаивает необработанной дольше ожидаемого. Задачи, не имеющие права на обработку Auto-merge, намеренно остаются In-review, пока человек не вмешается — проходы восстановления не должны их двигать.

### Merge queue (очередь слияния)
Упорядоченная очередь задач In-review, ожидающих Auto-merge, с единственным активным слиянием в каждый момент времени. Задачи входят только через гейты триггеров (проход при запуске движка, периодический повтор, снятие паузы и быстрый путь moved-to-review); задача, отфильтрованная на гейте, невидима для слияния независимо от её собственных настроек.

### Manual-required (требуется ручное)
Состояние запроса слияния для задачи, чьё слияние нуждается в явном человеческом разрешении — обычно задача с явно отключённым авто-слиянием под глобально включённым проектом. Достижение этого состояния требует, чтобы задача прошла через гейты триггеров Merge queue; вышестоящая фильтрация, исключающая такие задачи, оставляет их застрявшими в In-review вместо паркования здесь.

### Self-healing sweep (проход самовосстановления)
Повторяющееся фоновое сканирование, которое обнаруживает и чинит застрявшие состояния задач — застопорившиеся задачи In-review, подтверждённые слияния, так и не финализированные, призрачные или подвешенные состояния, исчерпанные повторы. Проходы уважают то же право на Auto-merge, что и Merge queue: они могут инспектировать любую задачу, но мутируют только те, что имеют право на обработку авто-слияния.

Проходы должны соблюдать те же правила цели слияния, что и нормальный путь — член Shared branch group всегда оценивается против ветки своей группы, никогда против ветки проекта по умолчанию — а атрибуция уже слитой работы должна якориться на маркерах владения коммитом, а не на совпадениях свободного текста.

### Shared branch group (группа общей ветки)
Когорта задач, интегрирующихся в общую разделяемую ветку, вместо того чтобы каждая сливалась напрямую в ветку проекта по умолчанию. Группа — а не её задачи-члены — владеет именем общей ветки, управляемой идентичностью PR и жизненным циклом группы (open, finalized, abandoned); члены ссылаются на свою группу по сохранённому id группы, никогда по выводимой строке. Интеграция члена (ветка задачи → общая ветка) — это мягкий шаг пред-интеграции, освобождённый от глобального гейта авто-слияния; продвижение группы (общая ветка → ветка по умолчанию) гейтится отдельно.

Общая ветка группы является только и всегда *целью* слияния; она никогда не является рабочей веткой какой-либо задачи-члена. Каждый член работает на собственной ветке на уровне задачи и приземляется на ветку группы.

### Branch assignment mode (режим назначения ветки)
Стратегия, по которой задача приобретает свою рабочую ветку и цель слияния. Режим shared даёт задаче рабочую ветку на уровне задачи, производную от общей ветки группы, и устанавливает общую ветку как цель слияния; режим per-task-derived даёт производную рабочую ветку без общей цели; остальные режимы (project default, existing, custom new) привязывают задачу напрямую к именованной ветке. Только режим shared создаёт членство в Shared branch group.

### Landed (приземлено)
Статус члена Shared branch group, чья работа подтверждённо слита на общую ветку *его собственной группы* через путь интеграции группы веток. Член, слитый на любую другую ветку — соседнюю ветку задачи, ветку проекта по умолчанию — не является Landed, независимо от его колонки. Группа завершена, когда у неё есть хотя бы один член и каждый член является Landed; завершённость гейтит продвижение группы.

### Group promotion (продвижение группы)
Гейтящийся завершённостью, идемпотентный акт переноса завершённой Shared branch group вперёд: слияние ветки группы в сторону интеграционной ветки проекта и, в режиме pull-request, создание-или-переиспользование единственного управляемого PR группы. Повторный запуск продвижения никогда не создаёт второй PR. При отключённом авто-слиянии продвижение является явным пользовательским действием; приземление члена-к-группе может всё равно продолжаться без его инициирования.

### PR entity (сущность PR)
Единая первоклассная запись pull request, которым управляет agi, независимо от того, как работа приземлилась — одиночная задача или Shared branch group каждая производит одну сущность PR с одним жизненным циклом (open, responding, approved, merged/closed). Это то, как фактически реализуется «управляемая идентичность PR» группы. Каждое состояние, которое показывает сущность, должно быть подтверждено GitHub; agi никогда не сохраняет спекулятивное состояние PR, а непрерывная реконсиляция поглощает внеполосные изменения, сделанные напрямую на GitHub.

### Review-response loop (цикл ответа на ревью)
Именованный процесс, посредством которого agi действует на обратную связь ревью, приходящую на сущность PR: новые комментарии и треды ревью от людей или ботов диспатчат агента, который либо чинит проблему, пушит в ветку PR и отвечает в треде, либо не соглашается — публикуя своё обоснование как комментарий PR и оставляя тред открытым. Работает без человеческого гейта между обратной связью и пушем; человеческая контрольная точка — это слияние (если не включено Auto-merge).

## Ветвление и атрибуция diff

### Integration branch (интеграционная ветка)
Локальная ветка (по умолчанию ветка проекта по умолчанию), где слияние приземляет ветки задач и от чьей вершины форкаются новые рабочие деревья задач. Поскольку слияние приземляет коммиты локально перед пушем, Integration branch может опережать свою копию на origin на слитые-но-незапушенные коммиты — любое вычисление точки форка или базы слияния должно измеряться сначала против локальной ветки, с ref origin только как откатом.

### Fork point (точка форка)
Коммит на Integration branch, из которого была создана ветка задачи — эксклюзивная нижняя граница изменений, которыми владеет задача. Каждое вычисление «изменённых файлов» делает diff от точки форка до вершины ветки, поэтому записанная база старше истинной Fork point навсегда приписывает файлы предшественников этой задаче.

### Rebase-and-push (ребейз-и-пуш)
Шаг после слияния, который ребейзит локально приземлённые коммиты слияния на вышестоящую ветку перед пушем, переписывая их SHA. Оригинальные коммиты становятся осиротевшими — больше недостижимыми с Integration branch — оставаясь при этом присутствующими в истории любой ветки задачи, форкнутой до пуша, поэтому слишком старая записанная Fork point не может быть восстановлена после этого шага.

### Contamination (загрязнение)
Чужеродные коммиты — работа, приписанная другим задачам — появляющиеся на ветке задачи за пределами её записанной Fork point. Проверки загрязнения должны вычислять свою опорную базу заново из Integration branch, а не переиспользовать сохранённую базу задачи, поскольку устаревшая сохранённая база заставляет каждый законно слитый коммит выглядеть чужеродным.

## Чат

### Generation (генерация)
Единый незавершённый оборот ассистента для чат-сессии, принадлежащий стороне сервера и идентифицируемый id генерации. Не более одной генерации работает на сессию: запуск новой прерывает любую генерацию, всё ещё активную для этой сессии, поэтому «лишняя» отправка от клиента никогда не безобидна. Генерация периодически сохраняет незавершённый снимок, чтобы переподключающийся клиент мог восстановить стримящийся UI.

### Queued message (поставленное в очередь сообщение)
Последующее сообщение, которое пользователь отправляет, пока генерация активна. Оно удерживается на стороне клиента (и сохраняется на сессию, чтобы навигация не могла его потерять) и сбрасывается — фактически отправляется — только когда генерация сессии устаканивается. Решения о сбросе должны приниматься против авторитетного состояния генерации сервера, а не локально закешированной копии.

### Enrichment field (поле обогащения)
Поле сессии, вычисляемое на маршруте API из живого состояния сервера (работает ли генерация, превью последнего сообщения), а не хранимое в строке сессии. Поля обогащения существуют только в ответах от обогащающих эндпоинтов: payload событий хранилища и SSE-вещания не имеют их, поэтому клиент, который перезаписывает своё состояние сессии из этих источников, молча деградирует поля обогащения до отсутствующих — любое решение с побочными эффектами, гейтящееся на одном из них, должно пере-запросить с обогащающего эндпоинта.

## Сессии Compound Engineering

### CE Stage (этап CE)
Зарегистрированный шаг конвейера compound-engineering (например, brainstorm, plan, work, compound), каждый сопоставленный с упакованным навыком и условным местоположением артефакта. Добавление этапа — это запись данных в реестр, а не новая кодовая поверхность.

### CE Session (сессия CE)
Единый интерактивный прогон CE Stage: агент управляет потоком вопрос/ответ с пользователем и производит артефакт этапа по завершении. Сессии — это независимые прогоны конвейера — множество может существовать одновременно, каждый со своим жизненным циклом (launching, active, awaiting-input, completed, error, interrupted) и историей беседы. Завершённая CE Session этапа work приземляет производные задачи на доску, со ссылкой обратно на сессию для происхождения.

### Detached turn (отсоединённый оборот)
Позиция исполнения для оборотов агента CE Session: запрос, инициирующий оборот, возвращается, как только сессия его отразит, а оборот работает в фоне, пока клиенты сходятся через push-события и опрос. Отсоединённый оборот никогда не отклоняется — каждый сбой сохраняется в состояние сессии и испускает наблюдаемое событие, поэтому прогресс никогда не теряется молча.

### Live activity (живая активность)
Транзиентный рабочий вывод незавершённого оборота агента — аккумулированное мышление, стримящийся текст и маркеры исполнения инструментов. Она наблюдаема, пока оборот работает, но не является состоянием сессии; когда оборот устаканивается или прерывается, сжатая трассировка сворачивается в историю беседы, чтобы транскрипт сохранял историю.

### Steering (направление)
Канал обратной связи пользователя в середине этапа: свободно-текстовое руководство, прикреплённое к ответу, или отправленное само по себе без ответа на ожидающий вопрос. Агенты трактуют направление как первоклассный ввод — учитывают его, корректируют курс и либо переспрашивают, либо продолжают.

### Rehydration (регидрация)
Повторное установление живого хэндла агента для приостановленной CE Session путём переигрывания её записанной беседы против модели. Переигрывание подавляет побочные эффекты: оно реконструирует контекст агента без повторного испускания событий, повторного стриминга Live activity или повторной записи артефактов.

## Плагины

### Bundled Plugin (упакованный плагин)
Плагин, который поставляется внутри самого дистрибутива agi, а не устанавливается из предоставленного пользователем пути — он появляется в Settings → Built-in Plugins и может быть автоматически установлен при запуске.
*Избегать:* built-in plugin (как отдельная концепция; метка в Settings использует «Built-in» для того же самого)

Bundled Plugin должен быть зарегистрирован в нескольких независимо поддерживаемых поверхностях — каталог Settings, fallback-набор bundled-id сервера дашборда, список автоустановки при запуске CLI и шаг сборки, который раскладывает загружаемую копию в дистрибутив. Поверхности не перекрёстно проверяют друг друга: плагин, зарегистрированный в некоторых, но не во всех, выглядит устанавливаемым, но не устанавливается или не загружается, поэтому добавление одного означает зеркалирование существующего упакованного плагина по каждой поверхности.

### Plugin Entry (точка входа плагина)
Единый загружаемый файл, сохраняемый как путь плагина и динамически импортируемый загрузчиком. Контракт строг: директория пакета никогда не является валидной точкой входа (ESM не может импортировать директории), поэтому каждая поверхность установки должна разрешить конкретный файл перед сохранением, предпочитая поставляемый бандл, затем предсобранный вывод, затем сырой исходник из workspace. Устаревшие регистрации, хранившие директорию, исцеляются на месте — перенаправляются на разрешённую точку входа — в следующий раз, когда плагин включается или автоустанавливается.

### Plugin Registry Entry (запись реестра плагинов)
Запись обнаружения дашборда, обслуживаемая из `GET /api/plugins/registry` и показываемая в Settings → Plugins → Browse Registry. Она метадата-ориентирована (`id`, `name`, `description`, `category`, опциональные version/author/homepage) и может включать конкретный `path` точки входа плагина.

### Registry Installability (устанавливаемость из реестра)
Выведенный сервером флаг `canInstall` для Plugin Registry Entry. `canInstall: true` означает, что запись манифеста имеет конкретный `path` и дашборд может вызвать нормальный флоу установки плагина; `canInstall: false` означает, что запись только для обнаружения и должна быть представлена как Coming Soon вместо попытки установки.

### Workflow Extension (расширение рабочего процесса)
Способность рабочего процесса, вносимая плагином, регистрируемая через движок, а не захардкоженная в ядро логики рабочего процесса: метадата колонки, политики перемещения, движки работы колонки, обработчики узлов рабочего процесса, провайдеры вердикта задачи или факты маршрутизации слияния. Workflow Extension является opt-in по метадате рабочего процесса и должно деградировать или парковаться по явной политике отката, когда его плагин отключён или отсутствует, сохраняя базовую линию рабочего процесса Default, когда никакое расширение не активно.

## Колонки и трейты рабочего процесса

*Контролируется включённым по умолчанию флагом `experimentalFeatures.workflowColumns`. С явным переопределением флага в выключенное состояние авторитетен и неизменен устаревший фиксированный конвейер (закрытый enum колонок + `VALID_TRANSITIONS`).*

### Column (workflow-defined) (колонка, определённая рабочим процессом)
Первоклассная, определённая рабочим процессом единица состояния задачи: id, отображаемое имя и набор конфигураций Trait. Позиция задачи на доске — это её текущая колонка, сохраняемая в `tasks."column"`. Валидность колонки ограничена рабочим процессом — устаревший закрытый enum расширяется до строки, валидируемой против разрешённого рабочего процесса задачи. Id колонок рабочего процесса Default байт-в-байт идентичны значениям устаревшего enum, поэтому ни одна строка задачи никогда не переписывается.

### Trait (трейт)
Композируемая конфигурация колонки: декларативные флаги (например, `complete`, `archived`, `countsTowardWip`) плюс опциональные хуки жизненного цикла (`guard`, `gate`, `onEnter`, `onExit`, `releaseCondition`). Встроенные и вносимые плагинами трейты регистрируются через один реестр. Синхронные хуки `guard` и флаги `complete`/`archived` доступны только встроенным; трейты плагинов получают только асинхронные точки хуков. Эффективные флаги колонки — это слитые флаги её трейтов; конфликтующие композиции отклоняются при сохранении (на стороне сервера и в редакторе).

### Column agent (агент колонки)
Постоянная привязка агента на определённой рабочим процессом колонке — агент из реестра плюс режим — укомплектовывающая всю работу с запуском сессии, атрибутируемую этой колонке (пользовательские узлы, сессия кодинга шва execute, сессии по шагам; узлы шаблона foreach наследуют колонку охватывающего foreach, если не объявляют свою собственную). `defer` делает агента колонки значением по умолчанию, применяясь только когда работа не несёт собственной идентичности агента и полной пары модели; `override` замещает настройки агента/модели на уровне узла и задачи целиком.

Требует и флага workflow-columns, и флага graph-executor; при любом выключенном привязки инертны во время исполнения. Отсутствующий или удалённый агент деградирует до нормального разрешения без прерывания живой сессии. Привязка агента, чья политика разрешений шире, чем значение по умолчанию проекта, требует явного подтверждения во время сохранения на каждой поверхности записи.

### Effective agent (execution principal) (эффективный агент — принципал исполнения)
Идентичность агента, которая фактически исполняет кусок работы после разрешения приоритета агента колонки — и принципал, с которым должна сверяться каждая подсистема, ключованная по идентичности: гейтинг разрешений, сериализация heartbeat в обоих направлениях, переотправка resume и обнаружение изменений в полёте. Она может отличаться от назначенного агента задачи под привязкой override, и одна задача может иметь несколько эффективных агентов через одновременные сессии веток.

### Lane (полоса)
Горизонтальный ряд на многополосной доске, по одному на рабочий процесс, используемый видимыми карточками. Каждая полоса рендерит колонки своего собственного рабочего процесса. Задачи без выбора рабочего процесса появляются в полосе рабочего процесса Default; каждая карточка появляется ровно в одной полосе. Полосы без карточек скрыты; полосы сворачиваемы с сохраняемым состоянием.

### Hold node (узел удержания)
Вид узла рабочего процесса, выражающий пассивное пребывание — карточка покоится в своей колонке, пока не сработает условие освобождения: ручное продвижение, таймер, доступная нижестоящая ёмкость, удовлетворённая зависимость или внешнее событие. Освобождение из удержания оценивается проходом субстрата (обобщённым планировщиком), который резервирует слоты рабочего дерева + семафора перед выдачей хода освобождения.

### Split / Join (разветвление / слияние)
Виды узлов параллельной ветки. `split` запускает свои исходящие рёбра одновременно; `join` синхронизирует их с `mode: all | any | quorum(n)` и `onBranchFailure: fail-fast | collect`. Во время параллельного окна карточка остаётся в колонке split-а (её позиция на доске никогда не форкается); при разрешении join она продвигается в колонку join-а. Узлы швов `execute`/`merge` запрещены внутри веток (одно рабочее дерево/сессия на задачу; слияние эксклюзивно). Состояние прогона по каждой ветке сохраняется в SQLite, поэтому упавшая ветка возобновляется с места, где умерла.

### Default workflow (рабочий процесс по умолчанию)
Встроенный рабочий процесс (`builtin:coding`), который дословно воспроизводит устаревший конвейер: шесть колонок, чьи id равны значениям устаревшего enum, с трейтами, соответствующими устаревшей семантике (`triage`=приём, `todo`=удержание+сброс-при-входе, `in-progress`=wip+прерывание-при-выходе+тайминг, `in-review`=блокировщик-слияния+обнаружение-застоя+слияние, `done`=complete, `archived`=archived). Null-выбор рабочего процесса разрешается в него во время чтения. Не редактируемый, не удаляемый.

### transitionPending (ожидающий переход)
Сохраняемый крэш-безопасный маркер (`tasks.transitionPending`), записываемый в той же транзакции, что и изменение колонки, фиксирующий пост-коммитные хуки (`hooksRemaining`), которые всё ещё должны идемпотентное исполнение. Очищается, как только они завершаются. Восстановление читает его исключительно из SQLite (авторитетного хранилища); крэш в середине перехода перезапускает идемпотентные хуки. Выбрасывающий исключение или отсутствующий хук деградирует (аудит) и очищает свою запись — он никогда не оставляет карточку застрявшей и не заклинивает блокировку задачи.

## Инверсия шагов

*Контролируется включённым по умолчанию флагом `experimentalFeatures.workflowGraphExecutor` (ортогональным `workflowColumns`). С явным переопределением флага в выключенное состояние, и для рабочего процесса Default всегда, политика шагов — это устаревший путь, принадлежащий движку (парсинг PROMPT.md, вердикты ревью в сессии, сброс RETHINK) — неизменный.*

### Step instance (экземпляр шага)
Одна рантайм-экспансия подграфа шаблона `foreach`, привязанная к единственному запланированному шагу (`Task.steps[i]`). Идентичность детерминирована — `<foreachNodeId>#<stepIndex>:<templateNodeId>` — поэтому resume реконструирует полный набор экземпляров из запиненного количества шагов без сохранения самой экспансии. Каждый экземпляр несёт собственное состояние прогона (текущий узел, счётчик переделок, baseline/checkpoint, а в режиме рабочего дерева — свою ветку и статус интеграции) в своей собственной сохраняемой таблице состояния прогона. Количество шагов запинено при экспансии; более позднее расхождение с живым списком шагов — это сбой `pin-mismatch`, никогда не молчаливая повторная экспансия. Записи жизненного цикла экземпляра протекают через `store.updateStep`, чтобы `Task.steps[]` оставался физическим проекционным стоком для каждого существующего потребителя.

### Artifact (артефакт)
Сохраняемая запись реестра, производимая агентами, чатом дашборда, рабочими процессами или задачами для переиспользуемых поставляемых результатов и промежуточных продуктов. Артефакты имеют тип (`document`, `image`, `video`, `audio` или `other`), атрибуцию автора, опциональную привязку к задаче, метадату, такую как MIME-тип/размер, и либо встроенный текстовый `content`, либо ссылку `uri`/path на хранимое медиа. Реестр артефактов хранит метадату для кросс-агентного обнаружения, тогда как дашборд показывает привязанные к задаче и не привязанные записи в галерее **Artifacts** представления Artifacts.

### parse-steps (парсинг шагов)
Узел графа рабочего процесса, который читает объявленный Artifact и запускает парсер из реестра для записи канонического списка шагов (`Task.steps[]`) — единственный графовый писатель шагов. Встроенные парсеры — это `step-headings` (конвенция `### Step N:`, извлечённая байт-в-байт из устаревшего regex, включая аннотацию `(depends: N,M)`) и `json-steps`; плагины вносят парсеры под `plugin:<pluginId>:<parserId>`. Сбои парсинга fail-closed маршрутизируются в `outcome:parse-error`, а не крэшат. Узел parse-steps должен доминировать (предшествовать на всех путях) любому `foreach(source:"task-steps")`, а запуск одного после того, как foreach уже экспандировал, срабатывает защиту пина (аудируемый сбой), чтобы циклы повторного планирования не могли рассинхронизировать экспандированную область.

### Custom task field (пользовательское поле задачи)
Объявленное рабочим процессом, типизированное поле задачи (`string | text | number | boolean | enum | multi-enum | date | url`, с опциями enum и подсказками рендеринга), чьи значения живут в `tasks.customFields`, ключованные по id поля. Модель задачи тем самым переосмысливается как ключевые поля (title, description) + стандартная метадата + эти определённые рабочим процессом поля. Записи проходят через единый авторитет хранилища (`updateTaskCustomFields`), который валидирует каждое значение против схемы разрешающего рабочего процесса и возвращает типизированные отклонения (нарушающий `fieldId` + `code`); агенты пишут их через патч `custom_fields` в `fn_task_update`. Редактирование полей рабочего процесса или переключение рабочего процесса задачи осиротевает (никогда не уничтожает) значения для удалённых или типово-несовместимых id — сироты сохраняются и показываются под раскрытием деталей, исключаются из карточек. Тот же id означает то же поле в рамках проекта; нет кросс-рабоче-процессного разделяемого пространства имён полей.

### Optional step group (опциональная группа шагов)
Контейнерный узел графа рабочего процесса (наряду с `foreach`/`loop`), чей подграф шаблона запускается один раз, когда задача его включила, и обходится в противном случае — графово-нативный способ сделать шаг опциональным на задачу. Включение — это переключатель на уровне задачи, засеянный из значения по умолчанию группы на уровне рабочего процесса; собственный id узла группы является ключом переключателя. Это заменяет более раннюю инертную для исполнения модель *объявления* (отдельный список опциональных шагов, прогоняемый через скрытый шов), поэтому опциональные шаги теперь реальные, размещаемые узлы, а не внеграфовая грань.

Единый проход — нет итерации или переделки внутри шаблона (это то, что отличает его от `foreach`/`loop`). Поскольку ключ переключателя — это id узла, переименование или пересоздание группы сбрасывает её включение на уровне задачи; и поскольку этот id может намеренно равняться id встроенного шаблона шага, набор включения на уровне задачи должен держать id групп стабильными по идентичности, а не прогонять их туда-обратно через устаревшую материализацию шаблона шага (которая переотобразила бы ключ и молча обошла бы группу).

## Персистентность и миграции

### Schema-Version Sweep (проход версии схемы)
Именованный процесс, выполняемый атомарно с любым повышением счётчика версии схемы ядра: репо-широкая охота за захардкоженными утверждениями старого номера версии, обновляемыми в том же коммите, что и повышение. Область прохода — каждое рабочее пространство, которое может встраивать базу данных ядра — пакеты *и* плагины — потому что любой пакет, инстанцирующий хранилище ядра, наблюдает текущую версию; ограничение охоты одним рабочим пространством молча оставляет утверждения застрявшими в других. Нижестоящие потребители должны предпочитать утверждение против экспортируемой константы версии вместо литерала, что удаляет их из прохода целиком.

## CLI executor

### CLI Executor (CLI-исполнитель)
Тип исполнителя `cli-agent`: сессия агента agi (шаг execute задачи, планирование, валидатор, сессия плагина CE или чат), управляемая интерактивным CLI-агентом кодинга, работающим в PTY, принадлежащем agi. Отличается от ранее существовавшего неинтерактивного вида исполнителя `cli` (раннер именованного скрипта/сырой команды). Выбирается на узле рабочего процесса для поверхностей задач (переопределение на уровне задачи) и на каждую сессию для чата/CE. Жизненный цикл доски неизменен — терминал является поверхностью исполнения, а не отдельным рабочим процессом.
*Избегать:* `cli` как идентификатор исполнителя — это имя занято видом раннера скриптов.

### CLI Adapter (CLI-адаптер)
Интеграция по каждому CLI, которая запускает и понимает один CLI-агент. Адаптеры нативной телеметрии (Claude Code, Codex, Droid, Pi) подключаются к собственным хукам/логам сессий CLI для точного состояния агента, структурированного транскрипта и нативной идентичности сессии; обобщённый адаптер запускает любую CLI-команду с эвристическим обнаружением простоя и представлением только сырого терминала. Адаптеры несут собственную конфигурацию запуска (команда, аргументы, режим разрешений) с поставляемыми значениями по умолчанию.

### CLI Session (CLI-сессия)
PTY, принадлежащий серверу, привязанный к сущности задачи или чата. Он переживает отключения клиентов, поддерживает одновременное подключение с любой поверхности и несёт состояние агента (starting, ready, busy, waiting-on-input, done, dead). Его нативный для CLI ID сессии сохраняется, поэтому мёртвый PTY или перезапуск движка возобновляется через собственный механизм resume CLI — needs-attention является откатом, когда resume проваливается, никогда не первым ответом.

### Waiting-on-input (ожидание ввода)
Состояние CLI Session, где агент заблокирован на человеке (запрос разрешения, уточняющий вопрос), в отличие от простоя-потому-что-выполнено. Вход в него инициирует уведомление, сконфигурированное на узле рабочего процесса; задача ни продвигается, ни проваливается, пока находится в нём.

## Тестирование

### Merge Gate (гейт слияния)
Минимальный набор блокирующих слияние проверок PR: lint, typecheck, build, Boot Smoke и небольшой курируемый набор тестов движка. Гейт — единственный тестовый сигнал, который может блокировать PR; все остальные тесты запускаются неблокирующе после слияния.

Членство в гейте — это явный allow-list, никогда не glob: тест зарабатывает своё место доказательством ценности и никогда не входит по умолчанию. Флейк внутри гейта *выселяется* — его запись в allow-list удаляется — что намеренно не требует зелёного прогона от самого флейкового теста, поэтому гейт всегда можно починить, пока он красный.

### Affected-package test selection (выбор тестов по затронутым пакетам)
Как `pnpm test` выбирает, что запускать против рабочего diff. В *changed mode* он запускает набор Merge Gate плюс пакеты workspace, которые diff затронул, и их зависимые; в *gate mode* он запускает только набор Merge Gate. Изменение в разделяемой тестовой инфраструктуре маршрутизируется в gate mode, который *заменяет* покрытие затронутых пакетов, а не добавляет к нему — поэтому diff, который срабатывает сигнал разделяемой инфры, но редактирует только файлы данных, не получает покрытия собственных пакетов, пока этот файл не внесён в allow-list как нерелевантный для тестов.

### Boot Smoke (дымовой тест запуска)
Доказательство гейта «приложение запускается и обслуживает»: CLI отвечает на свою команду help, а реальный сервер загружается на одноразовом порту, отвечает на свой эндпоинт health и чисто завершается по сигналу. Прохождение требует и того, что сигнал завершения был фактически доставлен, и того, что выход был чистым — крэш после обслуживания является проваленным путём загрузки, а не прохождением.

### Deletion Ratchet (храповик удаления)
Постоянная политика для флейковых тестов: тест, замеченный проваливающимся без соответствующего реального бага, помещается в карантин на месте — датированная запись в журнале плюс исключение из всех прогонов, не повторяется, не патчится — затем удаляется через 2 недели, если не спасён доказательством, что он ловит реальные регрессии, плюс исправлением первопричины. Умиротворение (расширенные таймауты, добавленные повторы, ослабленные утверждения) запрещено, особенно для агентов.

Второй карантин в той же подсистеме — это запах гонки продукта: флейк может быть реальным багом, поэтому код продукта получает взгляд до того, как истечёт часовой механизм удаления. Флейки гейта выходят через выселение из Merge Gate, а не через карантин, если только им не следует также покинуть неблокирующий уровень.

## Помеченные неоднозначности

- «Слияние» задачи из shared-branch-group использовалось как для интеграции члена, так и для продвижения группы — это различные шаги с независимым гейтингом, и их нельзя смешивать.
